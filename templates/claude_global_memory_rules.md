# Agent Memory（由 agent-mem0 管理）

**⚠ 以下是强制执行的工具调用规则，不是建议。每一条都必须在指定时机执行对应的 tool call，没有例外。如果你发现自己准备回复用户但还没有执行下面要求的 tool call，立即停下来先执行。**

## 规则 1：Session 首次响应前 — 必须 memory_search

**触发时机**：收到本 session 第一条用户消息时，在生成任何回复文本之前。

**执行动作**：调用 `memory_search`，query 使用用户消息的关键词。

**自检**：如果你正在写本 session 的第一条回复，但还没有调用过 `memory_search`，你正在违反规则。立即调用。

## 规则 2：涉及历史/上下文时 — 必须 memory_search

**触发条件**（任一成立即触发）：
- 用户消息含"之前"、"上次"、"继续"、"还记得"、"我们不是..."等引用历史的词
- 用户的问题涉及项目的业务逻辑、架构、历史决策
- 用户提到某个技术方案或决策
- 你对当前项目上下文有任何不确定

**唯一豁免**：纯通用技术问答（语法、标准库 API）且与当前项目完全无关。有疑问就搜。

## 规则 3：每轮回复结束前 — 评估记忆写入并选择正确工具

**触发时机**：每一轮回复的最后一步，在回复文本完成之后、提交之前。

**执行流程**：

**Step 1 — 是否有值得记录的信息？** 回顾本轮对话，检查是否发生以下任一情况：
- 做出了技术决策（选型、架构变更、方案确定）
- 完成了重要功能或修复
- 用户表达了偏好、习惯或工作方式
- 发现了重要问题、Bug 或临时方案
- 确定了重要的业务逻辑或规则
- 用户明确要求记住某些信息

如果没有命中任何一条，可以跳过本规则。

**Step 2 — 该信息是否可能与已有记忆冲突？** 如果命中以下任一场景，**必须先 `memory_search` 检查已有记忆**：
- 完成了修改现有架构、路径或配置的工作
- 推翻或修改了之前的技术决策
- 项目工作流或流程发生变化
- 修正了之前发现的 Bug 或问题（状态变更）
- 用户纠正了之前表达过的偏好

如果不属于上述场景（纯新增信息：全新功能、首次表达的偏好、新发现的外部资源），直接 `memory_add`。

**Step 3 — 根据 search 结果选择工具**：
- search 找到了相关但已过时的记忆 → **必须 `memory_update`**（传入 memory_id）
- search 未找到相关记忆 → `memory_add`

**不要写入**的：通用技术知识、单次调试细节、代码本身（代码在 git 里）。

**project 参数**：
- 与当前项目相关 → 使用当前项目名
- 与用户个人偏好/习惯相关 → `project="global"`

**自检**：如果你完成了修改已有架构/配置/决策的工作，但没有先 `memory_search` 检查是否有需要更新的记忆，你正在违反规则。补上 search，再决定 add 还是 update。

## 规则 4：用户要求忘记/删除记忆 — 必须 memory_search + memory_delete

**触发条件**：用户消息含"忘掉"、"删除记忆"、"不要记这个"、"forget"等删除意图的表达。

**执行流程**：
1. 调用 `memory_search` 定位目标记忆
2. 确认匹配后调用 `memory_delete`（传入 memory_id）

**保守原则**：agent 不得主动删除记忆。发现过时记忆时应使用 `memory_update` 修正，而非 delete。只有用户明确要求删除时才执行 delete。

**自检**：如果用户要求忘记某些信息，但你没有调用 `memory_delete`，你正在违反规则。

## 规则 5：用户要求查看记忆 — 必须 memory_list 或 memory_history

**触发条件与工具映射**：
- 用户要求列出/查看已有记忆（"看看记了什么"、"记忆列表"、"列出记忆"）→ 调用 `memory_list`
- 用户要求查看某条记忆的变更历史（"这条记忆的历史"、"记忆变更记录"）→ 调用 `memory_history`（传入 memory_id）

**自检**：如果用户明确要求查看记忆或记忆历史，但你没有调用对应工具而是凭记忆回答，你正在违反规则。
